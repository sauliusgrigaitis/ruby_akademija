24   #
 
25   # A message expectation is an expectation that the test double will receive a
 
26   # message some time before the example ends. If the message is received, the
 
27   # expectation is satisfied. If not, the example fails.
 
28   #
 
29   #   registrar = double("registrar")
 
30   #   validator.should_receive(:validate).with("02134")
 
31   #   zipcode = Zipcode.new("02134", validator)

33   #   
34   # When we declare a message expectation, we say we are "mocking" a method.

36   # == Mock Objects and Test Stubs
 
37   #
 
38   # The names Mock Object and Test Stub suggest specialized Test Doubles.  i.e.
 
39   # Test Stub evokes Test Double that only supports method stubs, and a Mock
 
40   # Object evokes a Test Double that only supports message expectations, or
 
41   # sometimes supports message expectations in addition to method stubs.
 
42   #
 
43   # There is a lot of overlapping nomenclature here, and there are many
 
44   # variations of these patterns (fakes, spies, etc). Keep in mind that most of
 
45   # the time we're talking about method-level concepts that are variations of
 
46   # method stubs and message expectations, and we're applying to them to _one_
 
47   # generic kind of object: a Test Double.
 
48   #
 
49   # == Test-Specific Extension
 
50   #
 
51   # a.k.a. Partial Stub/Mock, a Test-Specific Extension is an extension of a
 
52   # real object in a system that is instrumented with test-double like
 
53   # behaviour in the context of a test. This technique is very common in Ruby
 
54   # because we often see class objects acting as global namespaces for methods.
 
55   # For example, in Rails:
 
56   #
 
57   #   person = double("person")
 
58   #   Person.stub(:find) { person }
 
59   #
 
60   # In this case we're instrumenting Person to return the person object we've
 
61   # defined whenever it receives the +find+ message. We can do this with any
 
62   # object in a system because RSpec adds the +stub+ and +should_receive+
 
63   # methods to every object. When we use either, RSpec replaces the method
 
64   # we're stubbing or mocking with it's own test-double-like method. At the
 
65   # end of the example, RSpec verifies any message expectations, and then
 
66   # restores the original methods.
 
67   # 
68   # == Expecting Arguments
 
69   #
 
70   #   double.should_receive(:msg).with(*args)

74   #
 
75   # Arguments that are passed to #with are compared with actual arguments received
 
76   # using == by default. In cases in which you want to specify things about the arguments

115   #
 
116   # Whether you are setting a message expectation or a method stub, you can

118   # a block to +stub+ or +should_receive+:
